# soksak-plugin-git-workspace

soksak 워크트리 워크스페이스. 한 명령으로 브랜치와 git 워크트리를 만들고, 워크트리를 cwd 로 하는
터미널이 있는 전용 창을 열며, 열려 있는 워크스페이스를 목록으로 보여준다. 워크스페이스를 닫으면
워크트리 체크아웃과 창을 회수하지만 브랜치와 커밋은 그대로 남는다.

## 명령

- `worktree.open` — 워크트리 워크스페이스를 연다. 브랜치명 또는 이슈 슬러그를 받아 브랜치와 워크트리를
  생성하고(`soksak-spec-plugin-git` 구현체에 위임), 워크트리를 루트로 하는 프로젝트 창을 연다(기본 터미널이 그
  워크트리에 자리한다). 멱등: 이미 있는 워크스페이스를 열면 두 번째를 만들지 않고 그 창을 포커스한다.
- `worktree.close` — 워크트리 워크스페이스를 닫는다: 창을 정리하고 워크트리 체크아웃을 제거한다.
  브랜치와 커밋은 건드리지 않는다. 없는 워크스페이스는 no-op.
- `worktree.list` — 열려 있는 워크트리 워크스페이스 목록(뷰가 보여주는 것과 같은 레코드).

## 뷰

**워크스페이스** 패널(사이드바 또는 콘텐츠)이 열린 워크트리 워크스페이스를 나열한다. 각 행은 클릭 시
그 창을 포커스하고, 행별 컨트롤로 회수한다. 패널은 상태(loading / empty / active / error)를 뷰 status
축에 보고한다.

## 계약

`worktree.open`/`close`/`list` 는 git 을 실행하지 않는다. 필요한 git 연산(저장소 루트 판별, 브랜치
존재 확인, worktree add/list/remove)은 **`soksak-spec-plugin-git`** 에서 온다. 그 계약을 구현한 플러그인은
**계약의 identity 로 찾는다 — 이름으로 찾지 않는다**. `plugin.implementers` 를 버전-free 계약 id 를 담은
`{ id }` 로 호출한 뒤, 결과에서 활성 구현체를 고른다:

```sh
sok plugin.implementers '{"id":"soksak-spec-plugin-git"}'
```

발견 호출은 identity 만 싣는다 — 버전 range 를 넘기지 않는다. 매니페스트는
`consumes: [{ id: "soksak-spec-plugin-git", range: "0.0.1" }]` 를 선언한다 — 버전 range 가 사는 유일한
곳이며, 그래서 이 플러그인의 코드에도 매니페스트에도 구현체의 플러그인 id 가 없다. 구현체가 바뀌어도
이 플러그인은 그대로다. 활성 구현체가 없으면 loud 하게 거부한다(`NO_GIT_PROVIDER`) — 빈 워크스페이스
목록으로 답하지 않는다.

git 을 실행하지 않으므로 **`process` 권한을 갖지 않는다** — 아무것도 스폰할 수 없다. 그게 요점이다:
git 을 직접 돌리던 시절엔 ref 화이트리스트와 경로 증명을 자기 손으로 들고 있어야 했고, **방어의 중복은
보안 부채**다. 그 규칙들은 이제 계약에 있고, 거기서 채점된다.

창·터미널·패널 오케스트레이션은 코어 레지스트리 명령을 경유한다. 워크스페이스 레코드는 코어 데이터
스토어에 영속한다(`data` 권한).
